# 物理量・単位に関する方針

## 概要

XER では、物理量を型安全に扱うために、単位および次元を伴う量の仕組みを提供する。

この機能の目的は、C++ の既存ライブラリや将来の標準化提案をそのまま追従することではなく、XER 全体の方針に沿って、**軽量で分かりやすく、日常的に使いやすい物理量・単位機能**を提供することである。

XER ではすでに、時刻や環状値についても、標準ライブラリと完全に同一ではない独自設計を採っている。  
物理量・単位についても同様に、XER に適した簡潔な設計を採用する。

この文書は、物理量・単位機能の**初版方針文書**である。  
実装の細部は別途検討し、必要に応じて本方針文書へフィードバックする。

---

## 基本方針

### 目的

物理量・単位機能では、少なくとも次の点を実現する。

- 次元の異なる量どうしの誤加算・誤比較を防ぐ
- 単位変換を明示的かつ安全に行えるようにする
- 基本単位と組立単位を自然な記法で扱えるようにする
- 実用上よく使う範囲に絞り、過剰な一般化を避ける

### 内部表現

物理量は、**基準単位系に正規化した実数値**として内部保持する。

初期段階では、基準単位系は次の 4 つを採用する。

- 長さ: meter
- 質量: kilogram
- 時間: second
- 電流: ampere

すなわち、内部では実質的に **MKSA 系**を用いる。

たとえば次のように扱う。

- `1 km` は内部では `1000 m`
- `1 g` は内部では `0.001 kg`
- `1 msec` は内部では `0.001 sec`

内部保持値は常に基準単位での値とすることで、加減算・比較・乗除算・単位変換を単純化する。

## 次元の扱い

### 基本次元

初期段階で扱う基本次元は次の 4 つとする。

- 長さ
- 質量
- 時間
- 電流

温度、物質量、光度などは初期段階では扱わない。

### 次元表現

次元は、各基本次元に対する指数をテンプレート引数として持つ型で表現する。

```cpp
template <int L, int M, int T, int I>
struct dimension;
````

各引数の意味は次のとおりとする。

* `L`: length
* `M`: mass
* `T`: time
* `I`: electric current

### 例

* 長さ: `dimension<1, 0, 0, 0>`
* 質量: `dimension<0, 1, 0, 0>`
* 時間: `dimension<0, 0, 1, 0>`
* 電流: `dimension<0, 0, 0, 1>`
* 速度: `dimension<1, 0, -1, 0>`
* 力: `dimension<1, 1, -2, 0>`
* 電圧: `dimension<2, 1, -3, -1>`

### 無次元

無次元量は、全指数が 0 の次元で表現する。

```cpp
dimension<0, 0, 0, 0>
```

必要に応じて、次のような別名を設けてよい。

```cpp
using dimensionless = dimension<0, 0, 0, 0>;
```

## 物理量型

### 基本方針

物理量は、**数値**と**次元**を伴う値型として表現する。

```cpp
template <typename T, typename Dim>
class quantity;
```

ここで、

* `T` は保持値の型
* `Dim` は次元型

である。

### 保持値型

`quantity` の保持値型は、少なくとも初期段階では **浮動小数点型に制限する**。

想定する主な型は次のとおりである。

* `float`
* `double`
* `long double`

整数型は、少なくとも初期段階では `quantity` の保持値型として採用しない。

これは、次の理由による。

* 単位換算により小数値が自然に発生すること
* `rad` のような非有理数換算単位を扱うこと
* 基準単位への正規化後の内部値が整数に限られないこと
* 初期段階では設計を単純に保つことを優先すること

### 保持値

`quantity<T, Dim>` が保持する値は、**常に基準単位に正規化済みの値**とする。

たとえば、`quantity<float, time_dim>` に `0.001f` が入っていれば、それは `0.001 sec` を意味する。

### 生成と値取得

`quantity` は少なくとも次のような考え方で扱う。

* `quantity(T value)` は、**基準単位での値**から直接生成する
* `T * unit` により、指定単位から `quantity` を生成できる
* `value()` は基準単位での値を返す
* `value(unit)` は指定単位へ換算した値を返す

たとえば次のような利用を想定する。

```cpp
using namespace xer::units;

auto x = 1.5 * km;
auto y = x.value();    // 1500.0 (m)
auto z = x.value(km);  // 1.5
```

### 無次元量からの生値取得

無次元量であっても、実用上は生の数値として取り出したい場面がある。
そのため、無次元量から生の数値を取得する手段は提供する。

ただし、無次元量を暗黙に生の数値へ変換すると、型安全性が低下しやすい。
そのため、少なくとも初期段階では、**無次元量から生の数値への変換は明示的に行う**方針とする。

## 単位型

### 基本方針

単位は、**基準単位に対するスケール**と**次元**を表すものとして提供する。

少なくとも次の 2 要素を持つ。

* 基準単位に対するスケール
* 次元

ここでいう基準単位とは、初期段階では次の 4 つである。

* 長さ: `m`
* 質量: `kg`
* 時間: `sec`
* 電流: `A`

### `unit` の性格

`unit` は、できる限り **型として単位情報を表現するためのもの**とし、ランタイムに保持すべき実体データを持たない設計を優先する。

すなわち、単位に関する情報は原則としてテンプレート引数で表し、`unit` 自体のオブジェクトは空の値型として扱えることを目指す。

この方針は、次の理由による。

* 単位は値そのものではなく、物理量に付随する型情報として扱うのが自然であること
* 単位どうしの乗除算や合成を、可能な限りコンパイル時に解決したいこと
* `xer::units` 名前空間内の単位オブジェクトを軽量に保ちたいこと
* 不要なメモリ消費やランタイム処理を避けたいこと

したがって、`unit` は少なくとも初期段階では、**テンプレート引数で指定した内容だけを表し、追加のデータメンバは持たない形**を優先する。

### スケールの扱い

単位のスケールは、**有理数で表せるものに限らない**。

たとえば次のような単位は、有理数倍率で自然に表せる。

* `mm`
* `cm`
* `km`
* `g`
* `mg`
* `nsec`
* `microsec`
* `mA`
* `kHz`
* `GHz`
* `hPa`

一方で、角度単位 `rad` を `taurad` 基準で扱う場合、そのスケールは `1 / τ` となる。
これは有理数では表せない。

したがって、XER の単位機能では、**単位スケールを `std::ratio` 相当の有理数表現に固定しない**。

### スケール表現

単位スケールの表現は、少なくとも初期段階では **`std::ratio` 相当の有理数表現を基本**とする。

一方で、`rad` を `taurad` 基準で扱う場合のように、有理数では表せないスケールを持つ単位も存在する。
そのため、XER では **`std::ratio` を基本としつつ、必要に応じて浮動小数点ベースのスケール表現も扱える設計**とする。

少なくとも初期段階では、

* 通常の単位は `std::ratio` 相当で扱う
* 特殊な単位は `long double` 相当のスケール表現で扱う

という整理を採る。

### 単位演算におけるスケール

単位どうしの乗算・除算では、次元だけでなくスケールも合成される。

* 乗算: スケールを乗算する
* 除算: スケールを除算する

有理数単位どうしの演算結果は、可能な限り有理数表現のまま扱ってよい。
有理数で表せない場合は、浮動小数点ベースのスケール表現へ移行してよい。

## 単位オブジェクト

### 提供方法

ユーザーが日常的に使う単位は、`xer::units` 名前空間内の単位オブジェクトとして提供する。

```cpp
namespace xer::units {
    inline constexpr /* ... */ m = /* ... */;
    inline constexpr /* ... */ kg = /* ... */;
    inline constexpr /* ... */ sec = /* ... */;
    inline constexpr /* ... */ A = /* ... */;
}
```

単位オブジェクトは、ヘッダオンリー構成と相性が良いよう、`inline constexpr` とする。

### 名前空間

単位は `xer` 直下には置かず、専用の名前空間 `xer::units` にまとめる。

これにより、

* 単位名を通常 API と分離できる
* 必要な場面だけ `using namespace xer::units;` で取り込める
* `xer::msec` のような書き方を通常の記法にしなくて済む

### 利用例

```cpp
using namespace xer::units;

auto t = 1.23f * msec;
auto x = 10.0 * m;
auto i = 20.0 * mA;
```

必要であれば、完全修飾名も使えるものとする。

```cpp
auto t = 1.23f * xer::units::msec;
```

## 単位名の方針

### 基本方針

単位名は、可能な限り**普段よく使う記号表記**を採用する。

たとえば、次のような記号名を優先する。

* `m`
* `kg`
* `sec`
* `A`
* `V`

`meter`, `second`, `kilogram`, `volt` のような長い英単語は、少なくとも単位オブジェクト名としては採用しない。

### 秒について

秒については、単位オブジェクト名として `s` ではなく `sec` を採用する。

これは、

* 視認性を高めること
* `ms` などとの見分けを良くすること
* 一般コード中の識別子との衝突や読みにくさを避けること

を目的とする。

### 接頭辞について

SI 接頭辞のうち、マイクロについては `u` を用いない。

* 正式名として `micro` を用いる
* 必要に応じて `μ` を別名として用いてよい

したがって、たとえばマイクロ秒やマイクロアンペアを導入する場合は、少なくとも次のような形を優先する。

* `microsec`
* `microA`

また、必要に応じて次のような別名を導入してよい。

* `μsec`
* `μA`

ASCII 環境で `u` により代用されることが多いが、XER ではこれを正式表記として採用しない。

### `L` の表記について

体積単位 `L` については、`ℓ` を別名として採用しない。

## ユーザー定義リテラル

### 方針

ユーザー定義リテラルは定義しない。

検討の結果、次のような記法の方が自然であると判断する。

```cpp
1.23f * msec
```

そのため、物理量・単位機能では

* 単位オブジェクトを用いた乗算記法を基本とする
* ユーザー定義リテラルは導入しない

ものとする。

## 物理量演算

### 加減算

加減算は、**同一次元の量どうしに限り**許可する。

* `length + length` は可
* `length - length` は可
* `length + time` は不可

### 比較

比較も、**同一次元の量どうしに限り**許可する。

* `length < length` は可
* `length == length` は可
* `length < time` は不可

### 乗算・除算

乗算および除算は許可する。

このとき、結果の次元は指数の加減算で求める。

* 乗算: 次元指数を加算
* 除算: 次元指数を減算

### 生の数値との関係

`quantity * scalar`、`scalar * quantity`、`quantity / scalar` のような演算は許可してよい。
ただし、次元付き量と生の数値との暗黙変換は行わない。

### 例

* `m * m` → 面積
* `m / sec` → 速度
* `m / (sec * sec)` → 加速度
* `kg * m / (sec * sec)` → 力

---

## 単位演算

### 基本方針

単位どうしの乗算・除算は可能とする。

そのため、組立単位は必ずしも専用名を大量に用意しなくても表現できる。

### 例

```cpp
using namespace xer::units;

auto v = 3.0 * m / sec;
auto a = 9.8 * m / (sec * sec);
auto f = 2.0 * kg * m / (sec * sec);
```

この方針により、`m_sec` や `m_per_sec` のような合成単位を大量に定義する必要を避ける。

### 量と単位の混合演算

実装上は、少なくとも次のような記法を許可してよい。

* `quantity / unit`
* `scalar * unit`

これにより、たとえば `10.0 * m / sec` のような式を自然に書ける。

ただし、意味は常に **基準単位へ正規化した quantity を生成または変換すること** にある。

## 初期段階で提供する単位の選定基準

初期段階で標準提供する単位は、次のいずれかに該当するものを優先する。

* 日本の高校までで学習する単位
* ソフトウェアおよび電子工学の分野で頻出する単位
* 一般の書籍、報道、日常的な技術文書などで頻出する単位

一方で、次のような単位群は、少なくとも初期段階では標準提供を急がない。

* 特定分野への依存が強い単位
* 利用者層が限定的な単位
* 記号名が短く、他の識別子と衝突しやすい単位
* 系全体で整備しないと中途半端になりやすい単位群

たとえば CGS 系の単位については、一部分野で使われることは理解するが、少なくとも初期段階では XER 本体で広く抱え込まず、必要に応じて利用者側または分野別拡張で定義する方針とする。

## 角度の扱い

### 基本方針

角度は、一般の物理量と環状値とで役割を分ける。

* 周回数を含む通常の角度量は、**単純な実数値 + 単位**として扱う
* 環状性を伴う演算は、必要に応じて **`cyclic`** を用いる

XER の `cyclic` は、1 周を 1 とする環状値を扱うための型である。
そのため、角度量そのものをすべて `cyclic` で保持するのではなく、**環状演算のときだけ `cyclic` を使う**方が自然である。

### 例

* `2.25` 周
* `-0.1` 周
* `10.0` 周

のような量は、周回数を含む通常の量として保持できる方が望ましい。

一方、

* 1 周の中での位置
* 時計回り距離
* 反時計回り距離
* 最短差

のような環状概念については `cyclic` を用いる。

## 角度単位

### `rad` の扱い

`rad` は次元的には無次元である。
したがって、物理量・単位の枠組みにおいて `rad` を導入する場合、次元を増やすのではなく、**スケールの違いだけを持つ無次元単位**として扱う。

### `taurad` / `τrad`

XER では、`cyclic` が 1 周を 1 とする設計であることから、角度の基本単位として **`taurad`** を採用し、**`τrad`** を別名として提供する。

* `1 taurad` = 1 周
* `0.5 taurad` = 半周
* `0.25 taurad` = 1/4 周

この単位は `cyclic` との対応がきわめて明快である。

### 方針

角度単位については、少なくとも次のように扱う。

* 基本単位として `taurad`
* 別名として `τrad`

`rad` は **初期段階から提供する角度用の無次元単位**として扱う。

`taurad` を基本単位とするため、`rad` は `taurad` に対する換算単位として位置付ける。

### 周回数を含む角度量

周回数を含む角度量は、`cyclic` を内部保持するのではなく、**単純な実数値として保持し、必要な場面だけ `cyclic` に変換する**ものとする。

これにより、

* 角度の通常の加減算
* 角速度と時間からの角度計算
* 周回数の保持

といった用途を素直に扱える。

---

## `cyclic` との関係

`cyclic` は、角度そのものの一般表現ではなく、**環状演算のための補助型**として位置付ける。

したがって、角度量と `cyclic` の関係は次のように整理する。

* 角度量は通常の quantity として保持する
* 環状比較や最短差の計算時に `cyclic` へ変換する
* `cyclic` に単位を直接掛ける設計は採らない

初期段階では、物理量と `cyclic` との橋渡しのために、専用の高水準 API は必須とはしない。

少なくとも初期段階では、

* `quantity` から生の数値を明示的に取り出せること
* 生の数値から `quantity` を明示的に生成できること
* `cyclic` も同様に、生の数値を介して受け渡しできること

で十分とする。

これにより、利用者は必要に応じて

* 角度量から生値を取り出す
* その値を `cyclic` に渡す
* `cyclic` から得た値を再び角度量へ戻す

といった処理を自分で明示的に記述できる。

この方式は専用 API を用いる場合に比べて型安全性では劣るが、初期段階では設計を過度に複雑化しないことを優先する。

---

## 標準ライブラリとの関係

現行の標準ライブラリには、`std::chrono::duration` のような時間間隔の仕組みはあるが、一般的な物理量・単位ライブラリはまだ標準機能として確立していない。

一方で、数量・単位ライブラリの標準化提案は進んでいる。
ただし、XER ではそれをそのまま全面的に追うのではなく、初期段階では次の点を優先する。

* 軽量であること
* 分かりやすいこと
* XER 全体の設計と整合すること
* 過度に大規模なメタモデルを持ち込まないこと

したがって、初期段階では次のような高度な概念は見送ってよい。

* quantity kind
* quantity point
* 温度のような原点付き量
* 複雑な単位体系の一般化

## 暫定 API の考え方

現時点では、少なくとも次のような API 体系を想定してよい。

```cpp
template <int L, int M, int T, int I>
struct dimension;

using dimensionless = dimension<0, 0, 0, 0>;

template <typename Dim, typename Scale = std::ratio<1>>
class unit;

template <std::floating_point T, typename Dim>
class quantity;
```

また、少なくとも次のような利用を許容する。

```cpp
using namespace xer::units;

auto x = 1.5 * km;
auto t = 2.0 * sec;
auto v = x / t;

auto base = x.value();     // base unit value
auto kilo = x.value(km);   // value in km
```

## まとめ

* 物理量は、基準単位に正規化した値と次元を持つ
* 基本次元は、長さ・質量・時間・電流の 4 つとする
* 無次元は全指数 0 で表す
* `quantity` の保持値型は、少なくとも初期段階では浮動小数点型に制限する
* `quantity` は基準単位値を保持し、`value()` および `value(unit)` で値を取得する
* 単位は `xer::units` 名前空間内の `inline constexpr` オブジェクトとして提供する
* `unit` は実体データを持たない型レベルの表現を優先する
* ユーザー定義リテラルは定義しない
* 単位どうしの乗除算は可能とする
* 量と単位の混合演算により `10.0 * m / sec` のような自然な式を許可する
* 無次元量から生の数値への変換は、明示的に行う
* 単位スケールは有理数限定としない
* 通常単位は `std::ratio` 相当を基本とし、特殊単位は浮動小数点ベースのスケール表現も許容する
* 初期単位としては `m`, `kg`, `sec`, `A`, `Hz`, `N`, `J`, `W`, `V`, `Pa` を提供する
* 初期の接頭辞付き単位としては `mm`, `cm`, `km`, `microm`, `nm`, `g`, `mg`, `nsec`, `microsec`, `msec`, `mA`, `kHz`, `GHz`, `hPa` を提供する
* 面積・体積の慣用単位として `ha`, `mL`, `dL`, `L`, `kL` を提供する
* 熱量の慣用単位として `cal`, `kcal` を提供する
* マイクロ接頭辞には `u` を使わず、`micro` を正式名、`μ` を別名とする
* `cc` は `mL` の別名とする
* `ℓ` は `L` の別名として採用しない
* 角度は通常の量として保持し、環状演算のときだけ `cyclic` を使う
* 角度の基本単位として `taurad` を採用し、`τrad` を別名とする
* `rad` は初期段階から提供する角度用の無次元単位とする
* 物理量と `cyclic` の橋渡しは、少なくとも初期段階では生の数値の明示的な受け渡しで十分とする
