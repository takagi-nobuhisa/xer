# XER プロジェクト骨子

## 1. 目的

XER は、**X世代のC言語に慣れ親しんだプログラマ向けの C++23 ライブラリ**として設計する。  
C言語由来の分かりやすさを重視しつつ、C++23 の型安全性と表現力を取り入れる。  
過度に C++ 的な抽象化は避け、**C プログラマが理解しやすく扱いやすい**ことを優先する。

---

## 2. 対応環境

### 2.1 対応コンパイラ
- 当面の正式対象は **GCC 13.3.0 以上**とする
- 将来的に Visual C++ および Clang への対応を検討する
- 現時点では GCC のみを正式対象とする

### 2.2 対応プラットフォーム
- **Windows**
- **Linux**

### 2.3 Windows における対象環境
- コマンドプロンプト
- PowerShell
- MSYS2

MSYS2 では少なくとも以下を対象とする。

- `msys`
- `mingw64`
- `ucrt64`

### 2.4 Windows バージョン
- Windows 側は **Windows 11 以降**を対象とする

---

## 3. 提供形態

### 3.1 基本方針
- **ヘッダオンリ**とする
- インクルードするだけで利用できる形を目指す

### 3.2 ディレクトリ構成
- 公開ヘッダは `xer/` 配下に配置する
- 内部実装用の細分化した宣言・定義は `xer/bits/` 配下に配置する
- 公開ヘッダかどうかは拡張子ではなく **配置場所**で決める

### 3.3 ヘッダ拡張子
- ヘッダファイルの拡張子は **`.h`** とする

---

## 4. 名前空間と命名方針

### 4.1 名前空間
- 通常 API の名前空間は **`xer`**
- 低水準の上級者向け API は **`xer::advanced`**
- 内部実装用識別子は **`xer::detail`**

### 4.2 関数名
- 標準Cライブラリ再実装および PHP 関数再実装については、**元の関数名と同一の名前**を用いる
- ただし、同名でも C / PHP との完全互換を意味しない

---

## 5. C標準ライブラリ再実装の範囲

### 5.1 基本方針
- 再実装対象は **原則として C99 の関数群**とする
- ただし、必要なものから段階的に再実装する

### 5.2 優先順位
初期段階では以下を優先する。

1. **文字・文字列処理**
2. **入出力処理**
3. **JSON などの実用的なデータ形式処理**

数学関数は後回しとする。

### 5.3 対象外
少なくとも初期段階では以下を対象外とする。

- ロケール関連機能
- `inttypes.h`
- `fenv.h`
- `stdbool.h` など、C++ では不要な互換用ヘッダ

---

## 6. エンコーディング方針

### 6.1 対象エンコーディング
XER で扱う文字エンコーディングは、以下の 4 種類に限定する。

- **CP932**
- **UTF-8**
- **UTF-16**
- **UTF-32**

ロケールには依存しない。

### 6.2 文字と文字列
- **文字列**は原則として UTF-8 を用いる
- **文字**は必要に応じて UTF-32 を用いる
- 中間データやプラットフォーム接続では UTF-16 / UTF-32 も用いる
- 既存環境との互換のために CP932 も扱う

### 6.3 型との対応
- UTF-8 文字列: `char8_t`
- UTF-16 文字列: `char16_t`
- UTF-32 文字: `char32_t`
- CP932 文字列: `unsigned char`
- `char`: 基本表現には使わず、互換入力用とする

### 6.4 `char` 文字列の扱い
- `char` 文字列は内部の標準表現にはしない
- GCC 利用時は `-fexec-charset` を使わないことを前提とする
- ソースコードは **BOM 付き UTF-8** とする
- `char` 文字列のエンコーディング自動判定は通常運用には使わず、前提条件の検証用途に限定する

- `文字エンコーディング変換方針.md`

## 8. 入出力方針

### 8.1 基本方針
- 入出力は C 標準ライブラリの **`FILE` 系関数を土台**とした設計とする
- `iostream` は使用しない
- 公開 API は C++ ライブラリとして再設計し、C と完全互換の形は目指さない

### 8.2 公開型
公開 API では `FILE` をそのまま公開せず、用途に応じて以下の型を用いる。

- `binary_stream`
- `text_stream`

### 8.3 詳細参照

## 9. データ形式処理

### 9.1 基本方針
- C 標準ライブラリ由来ではないが、実用性の高いデータ形式処理も取り込む
- 初期実装では、JSON の encode / decode を `xer/json.h` として提供する
- JSON の内部文字列は UTF-8 とし、数値は `double` として扱う
- 通常の失敗は `xer::result` により返す

### 9.2 詳細
- `json_decode` は UTF-8 JSON 文字列を `json_value` に変換する
- `json_encode` は `json_value` を UTF-8 JSON 文字列に変換する
- object は順序保持のため、連想コンテナではなく要素列として保持する
- 初期実装では、コメント・末尾カンマ・`NaN`・`Infinity` などの拡張構文は扱わない

## 11. エラー処理方針

### 11.1 基本方針
- 例外は極力使用せず、通常の失敗は **`xer::result`** で表現する
- `xer::result<T, Detail>` の実体は **`std::expected<T, error<Detail>>`** とする
- ただし、比較不能や assert 系失敗など、設計上例外が適切な場面では例外を用いる

### 11.2 結果型
- XER における標準的な結果型として `xer::result<T, Detail = void>` を用いる
- `xer::result<T>` は `std::expected<T, error<void>>` の別名である
- `xer::result<T, Detail>` は `std::expected<T, error<Detail>>` の別名である
- 公開 API および内部実装では、失敗を返す関数の返却値型として原則 `xer::result` を用いる
- これにより、XER の返却型表記を簡潔にし、可読性と一貫性を高める

### 11.3 エラー型
- 詳細不要のケースでは `error<void>` を用いる
- `error<void>` は共通情報として `code` と `location` を持つ
- `message` メンバは持たない
- `error<Detail>` は追加情報を持つエラー型とする
- `Detail` がクラス型の場合、`error<Detail>` は `Detail` を継承する
- `Detail` が非クラス型の場合、`error<Detail>` は `detail` データメンバを持つ
- `Detail` はテンプレート引数で明示する

### 11.4 エラー生成
- エラー生成には `make_error` を用いる
- `std::source_location::current()` は `make_error` の引数既定値として最上位で取得する
- `error` のコンストラクタ側では `current()` を直接呼ばず、受け取った `location` を保持する
- `make_error(error_t)` は `error<void>` を生成する
- `make_error<Detail>(error_t, value)` は `error<Detail>` を生成する
- `value` は 1 個だけ受け取り、その値をそのまま `Detail` のコンストラクタまたは `detail` メンバ初期化に用いる
- 複数の値が必要な場合は、専用の集成体や `tuple` などで 1 個にまとめて渡す

### 11.5 `error_t`
- `error_t` はまず **`errno`** を基準に設計する
- 正値はターゲット環境の `errno`
- 負値はライブラリ独自エラー
- そのうち、入出力抽象層での失敗を表す共通コードとして `io_error` を設けてよい
- 成功値は持たない
- 基底型は当面 **`int32_t`** とする

### 11.6 assert 系
- コードのバグや内部不変条件違反については、通常の入出力エラーとは分離して扱う
- そのために XER 独自の **assert 系機構**を用意する
- 標準の `assert` のように異常終了させるのではなく、**失敗時は例外を送出**する

## 12. 共通ヘッダと処理系依存要素

### 12.1 共通ヘッダの役割
- 処理系前提の検証
- 言語バージョン前提の検証
- 文字コード前提の検証
- 処理系依存マクロ・拡張機能の吸収
- ライブラリ内部で使う共通抽象名の提供

### 12.2 検証方針
- 前提条件の確認は **共通ヘッダで常時行う**
- 検証は **`static_assert`** によりコンパイル時に行う
- 条件を満たさない場合はコンパイルエラーとする

### 12.3 共通マクロ
- C++ バージョン取得用の共通マクロ名は **`XER_STDCPP_VERSION`**
- 関数シグネチャ文字列取得用の共通マクロ名は **`XER_PRETTY_FUNCTION`**

## 13. テスト・コードジェネレーション方針

### 13.1 PHP の位置付け
- XER では、**コード生成およびテスト補助のために PHP を使用する**
- PHP は XER 本体の一部ではなく、**開発時専用ツール**として扱う

### 13.2 テストの分類
XER のテストは少なくとも以下に分類する。

- 正常系コンパイルテスト
- 異常系コンパイルテスト
- 実行テスト
