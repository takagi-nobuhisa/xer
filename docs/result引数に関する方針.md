# result引数に関する方針

## 概要

XER では、通常の公開 API において、`xer::result` を引数として受け取る設計は原則として採用しない。

従来は、エラー伝播を簡潔に記述できることを意図して、`xer::result` をそのまま受け取る関数も許容していた。
しかし、この方針は公開 API の見通しや使い勝手の面で不利に働く場合があることが分かった。

特に、通常の値を受け取るオーバーロードと `xer::result` を受け取るオーバーロードとが併存すると、呼び出し側で型推論やオーバーロード解決が不自然になりやすい。
その結果、文字列リテラルや一時オブジェクトを素直に渡せないなど、利用者にとって予想しにくい挙動を招くことがある。

このため、XER では `xer::result` 引数の利用範囲を限定し、公開 API をより単純で分かりやすいものに整理する。

---

## 基本方針

### 原則

通常の公開 API は、`xer::result` を引数として受け取らない。

関数は、あくまで通常の値を受け取って処理を行うものとする。
呼び出し側で `xer::result` を保持している場合は、呼び出し側が明示的に成功値を取り出してから渡す。

### 目的

この方針の目的は、主として次のとおりである。

- 公開 API の見通しを良くする
- オーバーロードの競合や曖昧性を避ける
- 文字列リテラルや一時オブジェクトを自然に渡せるようにする
- 利用者が関数の受理対象を直感的に理解しやすくする
- ドキュメントおよびコード例を単純化する

---

## 例外

### `xer/arithmetic.h`

`xer/arithmetic.h` に属する関数群については、例外として `xer::result` を引数に受け取ってよい。

これは、四則演算や比較補助において、複数段の演算結果をそのまま連鎖させたい需要が明確にあるためである。
特に、演算途中で発生したエラーをそのまま伝播したいという要件は、この領域では設計上の価値が大きい。

したがって、`xer::result` 引数の許容は、原則として `xer/arithmetic.h` に限定する。

### `xer::result` 自体を処理対象とする関数

`xer::result` そのものを処理対象とする目的の関数については、この方針の対象外とする。

たとえば、`xer::result` の成功・失敗を調べるための関数や、`xer::result` を変換・集約・補助的に処理するための関数については、当然ながら `xer::result` を引数に受け取ってよい。

これらは「通常の公開 API が便宜上 `xer::result` も受け取る」ケースとは性質が異なるためである。

---

## 設計上の考え方

### 公開 API は通常の値を受ける

通常の公開 API は、利用者がもっとも自然に書ける形を優先する。

たとえば、文字列処理関数であれば、`std::u8string_view` や `const char8_t*` 相当の入力を素直に受け取れることが重要である。
そこに `xer::result<std::u8string_view>` を受け取る便宜的なオーバーロードを加えると、使い勝手が悪化する場合がある。

そのため、公開 API においては、エラー伝播の都合よりも、通常入力に対する自然さを優先する。

### エラー伝播は呼び出し側で明示する

`xer::result` を保持している場合は、呼び出し側で明示的に成功値を取り出してから次の関数へ渡す。

これは多少記述量が増える場合もあるが、どの時点でエラーを確認し、どの値を次に渡しているのかが明確になる。
その結果、コードの意図が読みやすくなり、デバッグやドキュメント化も容易になる。

### 連鎖性が重要な領域だけ例外にする

演算系のように、連鎖と伝播そのものが API の価値になっている領域では、例外的に `xer::result` 引数を許容する。

ただし、この例外はあくまで限定的に扱う。
文字列処理、パス処理、入出力、時刻処理、JSON 処理など、通常の値を受けることが自然な API へは広げない。

---

## 既存 API の扱い

既存の公開 API のうち、`xer/arithmetic.h` 以外で `xer::result` を引数として受け取っているものは、順次廃止または削除する。

必要であれば、互換性に配慮して一時的な移行措置を設けてもよい。
ただし、alpha 段階であることを踏まえ、不要な互換層は極力持ち込まない。

---

## 今後追加する API の方針

今後追加する関数についても、原則として `xer::result` を引数に受け取らない。

新規 API を設計する際は、まず通常の値だけを受ける形を基本とし、本当に例外が必要な場合に限って個別に検討する。

このとき、「書けると便利そう」という程度の理由では `xer::result` 引数を採用しない。
採用する場合は、その関数領域において連鎖的なエラー伝播が公開仕様として本質的価値を持つことを必要条件とする。

---

## ドキュメントおよびコード例への影響

コード例およびリファレンスマニュアルでは、通常の値を関数へ渡す形を基本とする。

これにより、コード例は利用者がそのまま真似しやすいものとなる。
また、API 説明においても、受理する引数型を単純に記述できるようになる。

`xer::result` の扱いは、通常 API の引数仕様としてではなく、エラー処理の基本手段として説明する。

---

## まとめ

- 通常の公開 API は `xer::result` を引数に受け取らない
- `xer::result` 引数の許容は、原則として `xer/arithmetic.h` に限定する
- `xer::result` 自体を処理対象とする関数はこの方針の対象外とする
- 既存の公開 API のうち、`xer/arithmetic.h` 以外で `xer::result` を受け取るものは順次整理する
- 今後追加する関数も、原則として `xer::result` を引数に取らない
- 公開 API の自然さ、単純さ、分かりやすさを優先する
