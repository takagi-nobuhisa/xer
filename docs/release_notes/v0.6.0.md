# xer C++ Utility Library v0.6.0 Release Notes

xer C++ Utility Library v0.6.0 expands the library mainly in the areas of ZIP archive handling and fixed-schema binary serialization.

This release adds ZIP archive reading, writing, lookup, and extraction utilities through `<xer/zip.h>`. It also adds low-level binary serialization archives through `<xer/serialize.h>` and a PHP schema-based generator for C++ structures and `xfer` functions.

v0.6.0 also introduces a common `error_t::end_of_file` policy for sequential input APIs, keeping end-of-input distinct from lookup failure.

## Highlights

- Added `<xer/zip.h>` for ZIP archive utilities.
- Added `<xer/serialize.h>` for fixed-schema binary serialization.
- Added `php/generate_xfer_struct.php` for generating C++ structures and `xfer` functions from PHP schema arrays.
- Added `error_t::end_of_file` and updated sequential input APIs to use it for exhausted input.
- Added ZIP examples covering reading, creation, name lookup, and extraction.
- Added serialization examples covering hand-written `xfer`, generated single-structure schemas, and generated multiple-structure schemas.
- Updated README files, public header lists, reference manual fragments, policy documents, and examples to match the v0.6.0 API set.

## New Public Headers

### `<xer/zip.h>`

`<xer/zip.h>` provides ordinary ZIP archive handling as a compression-and-archive facility.

The main features are:

- opening ZIP archives for reading
- reading ZIP entry metadata sequentially
- locating entries by exact name
- reading complete entry bodies
- creating ZIP archives
- adding in-memory byte sequences as deflated entries
- adding source files as deflated entries
- explicitly finalizing writers with `zip_commit`
- extracting one entry to a selected path
- extracting one entry below a destination directory
- extracting all entries below a destination directory

The initial implementation targets ordinary non-ZIP64 single-disk archives. ZIP64, encrypted entries, archive comments, multi-disk archives, CP437 filename conversion, and streaming large-entry I/O are deferred.

### `<xer/serialize.h>`

`<xer/serialize.h>` provides low-level binary transfer archives for fixed-schema data.

The main entities are:

```cpp
class binary_output_archive;
class binary_input_archive;
```

The format is intentionally not self-describing. It stores data only, using fixed field order and little-endian scalar representation. Type names, field names, schema descriptions, object identifiers, byte-order markers, and version records are not written by the low-level archive layer.

Supported low-level field types include:

- `bool`
- fixed-width signed and unsigned integers
- `float`
- `double`
- `std::u8string`
- `std::vector<std::byte>`
- `std::array<T, N>`
- `std::vector<T>`
- `std::map<K, V>`

User-defined structures are handled through generated or hand-written `xfer` functions rather than reflection.

## ZIP Archive Utilities

### Reading Entries Sequentially

`zip_open` opens an archive for reading, and `zip_read` reads entry metadata from the central directory one entry at a time.

```cpp
auto archive = xer::zip_open(u8"sample.zip");
if (!archive) {
    return archive.error();
}

for (;;) {
    auto entry = xer::zip_read(*archive);
    if (!entry) {
        if (entry.error().code == xer::error_t::end_of_file) {
            break;
        }
        return entry.error();
    }

    auto name = xer::zip_entry_name(*entry);
    auto body = xer::zip_entry_read(*archive, *entry);
}
```

End of the entry stream is reported as `error_t::end_of_file`.

### Entry Lookup

Exact-name lookup is provided by `zip_locate_name`.

```cpp
auto entry = xer::zip_locate_name(archive, u8"docs/readme.txt");
```

Missing entries are reported as `error_t::not_found`. This keeps lookup failure distinct from sequential EOF.

`zip_entry_read_by_name` is a convenience function that performs lookup and reads the expanded entry body.

### Creating Archives

`zip_create` opens a ZIP archive for writing.

```cpp
auto archive = xer::zip_create(u8"output.zip");
```

Entries can be added from memory or from files:

```cpp
xer::zip_add_from_bytes(archive, u8"data.bin", bytes);
xer::zip_add_file(archive, u8"source.txt", u8"docs/source.txt");
```

Writers must be finalized explicitly:

```cpp
auto committed = xer::zip_commit(archive);
```

This is important because central-directory writing, flushing, and closing can fail.

### Extracting Archives

The extraction helpers are:

```cpp
auto zip_entry_extract(
    zip_archive& archive,
    const zip_entry& entry,
    std::u8string_view target_filename) -> xer::result<void>;

auto zip_entry_extract_to(
    zip_archive& archive,
    const zip_entry& entry,
    std::u8string_view destination_dir) -> xer::result<void>;

auto zip_extract_to(
    zip_archive& archive,
    std::u8string_view destination_dir) -> xer::result<void>;
```

`zip_entry_extract_to` and `zip_extract_to` apply path-safety checks to entry names. Empty names, absolute paths, drive-relative paths, NUL code units, empty interior components, and `.` or `..` components are rejected as `error_t::invalid_argument`.

## Fixed-Schema Binary Serialization

### Archive Model

The same `xfer` function can be used for output and input:

```cpp
struct record {
    std::uint32_t id;
    std::u8string name;
};

template<class Archive>
auto xfer(Archive& archive, record& value) -> xer::result<void>
{
    if (auto r = archive(value.id); !r) {
        return r;
    }
    if (auto r = archive(value.name); !r) {
        return r;
    }
    return {};
}
```

When the archive is `binary_output_archive`, fields are written. When the archive is `binary_input_archive`, fields are read into the same object.

### Binary Format

The format is compact and fixed:

- multi-byte scalar values are little-endian
- `bool` is stored as one byte, `0` or `1`
- `std::u8string` stores `uint64 byte_size` followed by UTF-8 bytes
- `std::vector<std::byte>` stores `uint64 byte_size` followed by raw bytes
- `std::array<T, N>` stores only elements; `N` is not stored
- `std::vector<T>` stores `uint64 element_count` followed by elements
- `std::map<K, V>` stores `uint64 element_count` followed by key/value pairs in map iteration order

The format is suitable when both sides share the same schema or are generated from the same schema source.

### Error Handling

Input-side data exhaustion is reported as `error_t::end_of_file`.

Invalid bool values and duplicate map keys are reported as `error_t::invalid_argument`. Excessively large length values are reported as `error_t::length_error`.

## PHP `xfer` Generator

`php/generate_xfer_struct.php` generates C++ structures and `xfer` functions from PHP schema arrays.

A single-structure schema can be written as:

```php
<?php

return [
    'namespace' => 'demo',
    'struct' => 'record',
    'fields' => [
        'id' => u32,
        'name' => s,
        'payload' => bin,
        'scores' => [[s, f64], m],
    ],
];
```

A multiple-structure schema can be written with `structs`:

```php
<?php

return [
    'namespace' => 'demo',
    'structs' => [
        'packet_header' => [
            'version' => u16,
            'kind' => u16,
        ],
        'payload' => [
            'name' => s,
            'values' => [f32, v],
        ],
    ],
];
```

The main type DSL tokens are:

```text
b
u8 u16 u32 u64
i8 i16 i32 i64
f32 f64
s
bin

[T, v]          // std::vector<T>
[T, [a, N]]     // std::array<T, N>
[[K, V], m]     // std::map<K, V>
```

The generator also supports `--generated-at=<text>` so examples and tests can produce reproducible generated headers.

## Common EOF Handling

v0.6.0 distinguishes sequential EOF from lookup failure.

- Sequential input exhaustion uses `error_t::end_of_file`.
- Named or keyed lookup failure uses `error_t::not_found`.

Examples:

- `zip_read` returns `end_of_file` when no more entries exist.
- `zip_locate_name` returns `not_found` when a named entry does not exist.
- stream and directory sequential reads use `end_of_file` for exhausted input.

## Notes and Limitations

The following items are intentionally deferred:

- ZIP64 support
- encrypted ZIP entries
- ZIP archive and entry comments
- multi-disk ZIP archives
- CP437 filename conversion
- streaming large-entry reads and writes
- self-describing serialization metadata
- automatic reflection-based user-defined-type serialization
- automatic schema compatibility management

These limitations are part of the initial v0.6.0 scope rather than accidental omissions.
