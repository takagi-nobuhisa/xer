# xer C++ Utility Library v0.6.0 リリースノート

xer C++ Utility Library v0.6.0 では、主に ZIP アーカイブ操作と固定スキーマのバイナリシリアライズを追加しました。

このリリースでは、`<xer/zip.h>` による ZIP アーカイブの読み取り、書き込み、検索、展開を追加しています。また、`<xer/serialize.h>` による低水準バイナリシリアライズアーカイブと、PHP スキーマ配列から C++ 構造体および `xfer` 関数を生成する仕組みを追加しています。

v0.6.0 では、逐次入力 API における `error_t::end_of_file` の扱いも整理し、入力終端と検索失敗を区別するようにしました。

## ハイライト

- `<xer/zip.h>` による ZIP アーカイブ操作を追加
- `<xer/serialize.h>` による固定スキーマのバイナリシリアライズを追加
- PHP スキーマ配列から C++ 構造体と `xfer` 関数を生成する `php/generate_xfer_struct.php` を追加
- `error_t::end_of_file` を追加し、逐次入力 API で入力終端を表すように整理
- ZIP の読み取り、作成、名前検索、展開のコード例を追加
- 手書き `xfer`、単一構造体生成、複数構造体生成のシリアライズコード例を追加
- README、公開ヘッダ一覧、リファレンスマニュアル断片、方針文書、コード例を v0.6.0 API に合わせて更新

## 新しい公開ヘッダ

### `<xer/zip.h>`

`<xer/zip.h>` は、ZIP アーカイブを圧縮・アーカイブ操作のための機能として提供します。

主な機能は次のとおりです。

- ZIP アーカイブを読み取り用に開く
- ZIP エントリメタデータを逐次読み取る
- エントリを正確な名前で検索する
- エントリ本体全体を読み取る
- ZIP アーカイブを作成する
- メモリ上のバイト列を deflate エントリとして追加する
- 元ファイルを deflate エントリとして追加する
- `zip_commit` で書き込み側を明示的に最終化する
- 1つのエントリを指定パスへ展開する
- 1つのエントリを展開先ディレクトリ配下へ展開する
- すべてのエントリを展開先ディレクトリ配下へ展開する

初期実装の対象は、通常の非 ZIP64・単一ディスク ZIP アーカイブです。ZIP64、暗号化エントリ、アーカイブコメント、マルチディスクアーカイブ、CP437 ファイル名変換、大きなエントリのストリーミング I/O は後回しです。

### `<xer/serialize.h>`

`<xer/serialize.h>` は、固定スキーマデータのための低水準バイナリ転送アーカイブを提供します。

主なエンティティは次のとおりです。

```cpp
class binary_output_archive;
class binary_input_archive;
```

この形式は意図的に自己記述型ではありません。固定されたフィールド順序とリトルエンディアンのスカラ表現でデータだけを格納します。型名、フィールド名、スキーマ説明、オブジェクト識別子、バイトオーダーマーカー、バージョンレコードは低水準アーカイブ層では書き込みません。

低水準で対応するフィールド型は次のとおりです。

- `bool`
- 固定幅の符号付き整数・符号なし整数
- `float`
- `double`
- `std::u8string`
- `std::vector<std::byte>`
- `std::array<T, N>`
- `std::vector<T>`
- `std::map<K, V>`

ユーザー定義構造体は、リフレクションではなく、生成または手書きの `xfer` 関数で扱います。

## ZIP アーカイブ操作

### エントリの逐次読み取り

`zip_open` はアーカイブを読み取り用に開き、`zip_read` は中央ディレクトリからエントリメタデータを1つずつ読み取ります。

```cpp
auto archive = xer::zip_open(u8"sample.zip");
if (!archive) {
    return archive.error();
}

for (;;) {
    auto entry = xer::zip_read(*archive);
    if (!entry) {
        if (entry.error().code == xer::error_t::end_of_file) {
            break;
        }
        return entry.error();
    }

    auto name = xer::zip_entry_name(*entry);
    auto body = xer::zip_entry_read(*archive, *entry);
}
```

エントリ列の終端は `error_t::end_of_file` として報告されます。

### エントリ検索

正確な名前による検索は `zip_locate_name` で行います。

```cpp
auto entry = xer::zip_locate_name(archive, u8"docs/readme.txt");
```

エントリが見つからない場合は `error_t::not_found` として報告されます。これにより、検索失敗と逐次 EOF を区別できます。

`zip_entry_read_by_name` は、検索と展開済みエントリ本体の読み取りをまとめて行う便利関数です。

### アーカイブ作成

`zip_create` は ZIP アーカイブを書き込み用に開きます。

```cpp
auto archive = xer::zip_create(u8"output.zip");
```

エントリはメモリまたはファイルから追加できます。

```cpp
xer::zip_add_from_bytes(archive, u8"data.bin", bytes);
xer::zip_add_file(archive, u8"source.txt", u8"docs/source.txt");
```

書き込み側は明示的に最終化する必要があります。

```cpp
auto committed = xer::zip_commit(archive);
```

中央ディレクトリの書き込み、flush、close が失敗する可能性があるため、これは重要です。

### アーカイブ展開

展開ヘルパーは次のとおりです。

```cpp
auto zip_entry_extract(
    zip_archive& archive,
    const zip_entry& entry,
    std::u8string_view target_filename) -> xer::result<void>;

auto zip_entry_extract_to(
    zip_archive& archive,
    const zip_entry& entry,
    std::u8string_view destination_dir) -> xer::result<void>;

auto zip_extract_to(
    zip_archive& archive,
    std::u8string_view destination_dir) -> xer::result<void>;
```

`zip_entry_extract_to` と `zip_extract_to` は、エントリ名に対してパス安全性チェックを行います。空の名前、絶対パス、ドライブ相対パス、NUL コード単位、中間の空要素、`.` または `..` 要素は `error_t::invalid_argument` として拒否されます。

## 固定スキーマのバイナリシリアライズ

### アーカイブモデル

同じ `xfer` 関数を出力と入力の両方に使用できます。

```cpp
struct record {
    std::uint32_t id;
    std::u8string name;
};

template<class Archive>
auto xfer(Archive& archive, record& value) -> xer::result<void>
{
    if (auto r = archive(value.id); !r) {
        return r;
    }
    if (auto r = archive(value.name); !r) {
        return r;
    }
    return {};
}
```

アーカイブが `binary_output_archive` の場合はフィールドを書き込みます。`binary_input_archive` の場合は同じオブジェクトのフィールドへ読み込みます。

### バイナリ形式

形式はコンパクトで固定されています。

- 複数バイトのスカラ値はリトルエンディアン
- `bool` は1バイトで、`0` または `1`
- `std::u8string` は `uint64 byte_size` に続けて UTF-8 バイト列
- `std::vector<std::byte>` は `uint64 byte_size` に続けて生バイト列
- `std::array<T, N>` は要素だけを格納し、`N` は格納しない
- `std::vector<T>` は `uint64 element_count` に続けて要素列
- `std::map<K, V>` は `uint64 element_count` に続けて map の反復順で key/value ペア

この形式は、双方が同じスキーマを共有している場合、または同じスキーマソースから生成されている場合に適しています。

### エラー処理

入力側でデータが不足している場合は `error_t::end_of_file` として報告されます。

不正な bool 値や重複した map キーは `error_t::invalid_argument` として報告されます。過度に大きな長さ値は `error_t::length_error` として報告されます。

## PHP `xfer` ジェネレータ

`php/generate_xfer_struct.php` は、PHP スキーマ配列から C++ 構造体と `xfer` 関数を生成します。

単一構造体スキーマは次のように書けます。

```php
<?php

return [
    'namespace' => 'demo',
    'struct' => 'record',
    'fields' => [
        'id' => u32,
        'name' => s,
        'payload' => bin,
        'scores' => [[s, f64], m],
    ],
];
```

複数構造体スキーマは `structs` で書けます。

```php
<?php

return [
    'namespace' => 'demo',
    'structs' => [
        'packet_header' => [
            'version' => u16,
            'kind' => u16,
        ],
        'payload' => [
            'name' => s,
            'values' => [f32, v],
        ],
    ],
];
```

主な型 DSL トークンは次のとおりです。

```text
b
u8 u16 u32 u64
i8 i16 i32 i64
f32 f64
s
bin

[T, v]          // std::vector<T>
[T, [a, N]]     // std::array<T, N>
[[K, V], m]     // std::map<K, V>
```

ジェネレータは `--generated-at=<text>` もサポートします。これにより、コード例やテストで再現可能な生成ヘッダーを作れます。

## EOF の共通化

v0.6.0 では、逐次 EOF と検索失敗を区別します。

- 逐次入力の終端は `error_t::end_of_file`
- 名前やキーによる検索失敗は `error_t::not_found`

例:

- `zip_read` は、これ以上エントリがない場合に `end_of_file` を返します。
- `zip_locate_name` は、指定名のエントリがない場合に `not_found` を返します。
- ストリームやディレクトリの逐次読み取りは、入力終端を `end_of_file` として扱います。

## 注意点と制限事項

次の項目は意図的に後回しにしています。

- ZIP64 対応
- 暗号化 ZIP エントリ
- ZIP アーカイブコメントとエントリコメント
- マルチディスク ZIP アーカイブ
- CP437 ファイル名変換
- 大きなエントリのストリーミング読み書き
- 自己記述型のシリアライズメタデータ
- リフレクションベースのユーザー定義型自動シリアライズ
- スキーマ互換性の自動管理

これらの制限は、偶然の抜けではなく、初期 v0.6.0 の対象範囲として意図的に残しているものです。
