# xer C++ Utility Library v0.8.0 Release Notes

xer C++ Utility Library v0.8.0 focuses on practical conversion, local communication support, quantity units, and diagnostic usability. This release adds the generic `xer::to<T>` conversion entry point, completes length-prefixed socket message helpers, expands quantity units with international yard-pound units, and adds the lightweight `xer_print` diagnostic helper for examples and quick checks.

This release also reorganizes unit implementation headers, improves examples and tests, and updates the documentation set for the v0.8.0 API surface.

## Highlights

- Added `<xer/convert.h>` with the generic `xer::to<T>` conversion function.
- Added range-checked numeric conversion based on `xer::in_range`.
- Added explicit text conversion support for `char8_t`, `char16_t`, `char32_t`, and `wchar_t` strings.
- Deliberately avoids interpreting ambiguous narrow `char` strings as text in `xer::to<T>`.
- Extended `<xer/socket.h>` with address-specific bind, exact byte transfer helpers, and length-prefixed message send/receive helpers.
- Added international yard-pound length and mass units to `<xer/quantity.h>`.
- Split predefined SI/common units and yard-pound units into separate internal implementation headers.
- Added `xer_print` to `<xer/diag.h>` for simple expression printing and result display.
- Cleaned up examples and tests by replacing repeated local helper functions with shared facilities where appropriate.
- Updated README files, public header lists, reference manual fragments, Japanese reference fragments, translation terminology, and policy documents for the v0.8.0 API set.

## New Public Header

### `<xer/convert.h>`

`<xer/convert.h>` provides the generic `xer::to<T>` conversion function:

```cpp
template <class To, class From>
auto to(From&& from) -> xer::result<To>;
```

Unlike a C++ cast expression, `xer::to<T>` may parse text, format values, validate numeric ranges, normalize selected xer value types, and fail. Therefore it returns `xer::result<T>`.

Typical examples include:

```cpp
auto n = xer::to<int>(u8"123");
auto s = xer::to<std::u8string>(123);
auto p = xer::to<xer::path>(u8"dir\\file.txt");
```

Numeric-to-numeric conversions are checked with `xer::in_range` before conversion. This prevents accidental narrowing or signed/unsigned range mistakes:

```cpp
xer::to<unsigned char>(255); // success
xer::to<unsigned char>(256); // error_t::range
xer::to<unsigned>(-1);       // error_t::range
```

`char` is treated as a character type, while `signed char` and `unsigned char` are treated as integer types.

`xer::to<T>` supports explicitly encoded text strings using `char8_t`, `char16_t`, `char32_t`, and `wchar_t`. Ambiguous narrow `char` strings are intentionally not interpreted as text by `xer::to<T>`, so calls such as `xer::to<int>("123")` and `xer::to<std::u8string>("abc")` are rejected. This avoids silently treating platform-dependent narrow strings as UTF-8, locale text, or native path text.

## Socket Updates

`<xer/socket.h>` now provides additional helpers for small TCP-based local protocols and similar use cases.

New or expanded operations include:

- `socket_bind(socket&, address, port)` for binding to a specific local address such as `127.0.0.1`
- `socket_send_all` for sending an entire byte span over a stream socket
- `socket_recv_exact` for receiving an exact number of bytes from a stream socket
- `socket_send_message` for sending a 4-byte big-endian length-prefixed message
- `socket_recv_message` for receiving a length-prefixed message with a caller-supplied maximum size

The length-prefixed message format is:

```text
uint32 big-endian payload_size
payload bytes
```

`socket_recv_message` rejects messages larger than the supplied maximum size and does not discard the oversized payload. Callers should close the connection after such a protocol error.

These helpers are useful for configurations such as an HTTP/PHP API layer communicating with a local persistent control process over TCP localhost.

## Quantity and Unit Updates

`<xer/quantity.h>` now includes predefined international yard-pound units in addition to the existing SI and common metric units.

The added unit objects are:

- `inch`
- `ft`
- `yd`
- `mile`
- `oz`
- `lb`

They are provided under `xer::units` and use exact international definitions relative to SI units where applicable:

- `1 inch = 0.0254 m`
- `1 ft = 0.3048 m`
- `1 yd = 0.9144 m`
- `1 mile = 1609.344 m`
- `1 lb = 0.45359237 kg`
- `1 oz = 1/16 lb`

The public include model remains unchanged:

```cpp
#include <xer/quantity.h>
```

Internally, predefined unit definitions are now split into implementation headers such as `xer/bits/units_si.h` and `xer/bits/units_imperial.h`. This is an implementation detail and does not make those headers public.

## Diagnostic Printing

`<xer/diag.h>` now provides `xer_print` for simple diagnostic output:

```cpp
xer_print(expression);
xer_print(label, expression);
```

The one-argument form uses the source expression text as the label. The two-argument form uses an explicit UTF-8 label.

`xer_print` formats values through xer's `%@` printf conversion. For `result<T>`, a successful result prints the contained value and a failed result prints `error(name)`. For `result<void>`, a successful result prints `ok`.

`xer_print` is intended for examples, quick checks, and lightweight diagnostics. It is separate from `xer_trace`, which remains the category- and level-based trace facility and is disabled when `NDEBUG` is defined.

## Examples and Tests

Examples were cleaned up to use `xer_print` for simple display instead of repeatedly defining local `print_result`, `print_line`, or similar helper functions.

Tests were also reorganized with shared test helper utilities for common file, byte, and temporary-directory operations. This reduces repeated local support code while keeping the tests focused on the behavior being checked.

These changes are internal to the repository's examples and tests and do not affect the public API.

## Compatibility Notes

v0.8.0 adds one new public header and several new functions. Existing v0.7.0 public APIs remain within the documented v0.x stability policy.

The most important behavioral note is that `xer::to<T>` intentionally does not interpret narrow `char` strings as text. Use explicitly encoded string literals such as `u8"..."`, `u"..."`, `U"..."`, or `L"..."` when requesting text parsing or text conversion.

Internal unit definition headers were split for maintainability, but users should continue including `<xer/quantity.h>` rather than including headers under `xer/bits/` directly.
