# xer C++ Utility Library v1.1.0 Release Notes

xer C++ Utility Library v1.1.0 is a toolchain and test-infrastructure release.

This release does not add a new feature area. Instead, it expands the environments in which xer can be built and tested, with a particular focus on Clang-based toolchains and Visual Studio 2026. The same test programs, examples, and public-header pair checks are now available across GCC, Clang, MSYS2 CLANG64, Visual Studio clang-cl, and Visual Studio MSVC cl.exe environments.

## Highlights

- Added support for Ubuntu Clang with libc++.
- Added support for MSYS2 CLANG64.
- Added support for Visual Studio 2026 with clang-cl.
- Added support for Visual Studio 2026 with MSVC cl.exe.
- Updated `php/run_tests.php` to support explicit toolchain selection for GCC, Clang, clang64, clang-cl, and MSVC.
- Updated `php/test_public_header_pairs.php` to support the same toolchain selection model.
- Added vcpkg manifest-mode based detection for ICU and zlib on Visual Studio 2026.
- Improved Windows Tcl/Tk detection, with Tcl/Tk 9.0 preferred over 8.6.
- Cleaned up Windows API, Winsock2, `NOMINMAX`, and `min` / `max` macro handling.
- Reworked Windows directory enumeration around `FindFirstFileExW` and `FindNextFileW`.
- Added MSVC handling for `/Zc:__cplusplus` and `/Zc:preprocessor`.
- Added safer handling for environments where `__int128` is not used as a regular xer feature.
- Improved test failure diagnostics by writing uncaught-exception information to `run.log`.
- Moved MSVC / clang-cl object-file output under `php/build/`.
- Excluded `vcpkg_installed/` from Git and release ZIP packages.
- Updated README files, policy documents, and reference fragments for the expanded toolchain support.

## Expanded Toolchain Support

v1.1.0 expands the officially supported and tested toolchain set.

The current compiler targets include:

- GCC 13.3.0 or later
- Clang 18.0.0 or later
- Visual Studio 2026 clang-cl
- Visual Studio 2026 MSVC cl.exe

The tested environments include:

- Ubuntu with GCC
- Ubuntu with Clang and libc++
- MSYS2 UCRT64 with GCC
- MSYS2 CLANG64 with Clang
- Visual Studio 2026 with clang-cl
- Visual Studio 2026 with MSVC cl.exe

The MSVC cl.exe support uses the same test and example coverage as the other supported toolchains.

When using MSVC cl.exe directly, xer requires the following conformance options:

```bat
/Zc:__cplusplus /Zc:preprocessor
```

`/Zc:__cplusplus` is required so that `__cplusplus` reports the selected C++ language mode correctly.
`/Zc:preprocessor` is required for standard-conforming variadic macro expansion used by xer diagnostic macros.

The bundled PHP test scripts add these options automatically when `--tc=msvc` is specified.

## Test Script and Build Infrastructure Updates

`php/run_tests.php` and `php/test_public_header_pairs.php` were updated so that tests can be run explicitly with different toolchains.

Typical examples:

```sh
php run_tests.php tests examples --jobs=8 --all --tc=gcc
php run_tests.php tests examples --jobs=8 --all --tc=clang
php run_tests.php tests examples --jobs=8 --all --tc=clang64
php run_tests.php tests examples --jobs=8 --all --tc=clang-cl
php run_tests.php tests examples --jobs=8 --all --tc=msvc
```

Public header pair checks also support toolchain selection:

```sh
php test_public_header_pairs.php --jobs=8 --tc=gcc
php test_public_header_pairs.php --jobs=8 --tc=clang
php test_public_header_pairs.php --jobs=8 --tc=clang64
php test_public_header_pairs.php --jobs=8 --tc=clang-cl
php test_public_header_pairs.php --jobs=8 --tc=msvc
```

Build outputs are separated by toolchain under `php/build/`, including:

```text
php/build/linux-gcc/
php/build/linux-clang/
php/build/windows-msys2-ucrt64/
php/build/windows-msys2-clang64/
php/build/windows-msvc-clang-cl/
php/build/windows-msvc-cl/
```

Object files generated by MSVC and clang-cl are also written under the corresponding build directory instead of the `php` directory.

## Windows and Visual Studio 2026 External Dependencies

v1.1.0 adds Visual Studio 2026 support for optional external dependencies.

The repository includes `vcpkg.json`. ICU and zlib can be installed from the project root with:

```bat
vcpkg install --triplet x64-windows
```

In manifest mode, vcpkg installs dependencies under:

```text
vcpkg_installed\x64-windows
```

`run_tests.php` detects the corresponding include, lib, and bin directories and applies them to compiler include paths, linker libraries, and runtime DLL search paths.

Tcl/Tk detection on Windows was also improved. Tcl/Tk 9.0 is preferred over 8.6. If automatic detection is not sufficient, the installation root can be specified with `XER_TEST_TCLTK_ROOT`:

```bat
set XER_TEST_TCLTK_ROOT=C:\local\IronTcl
```

Tcl/Tk distributions such as Magicsplat Tcl/Tk, ActiveTcl, and other Windows binary distributions can be used if they provide the required headers, import libraries, and runtime DLLs.

## Windows API and Winsock2 Cleanup

Windows header handling was cleaned up as part of the Visual Studio 2026 work.

The main changes include:

- cleaned up the include order around `winsock2.h` and `windows.h`
- added `NOMINMAX` handling
- added protection against `min` / `max` macro conflicts
- separated ordinary Windows API usage from socket-specific Windows header usage
- reworked Windows directory enumeration to use `FindFirstFileExW` and `FindNextFileW`

This reduces conflicts when `<xer/socket.h>` is used together with other Windows-related headers.

---

## Handling Environments Without `__int128`

MSVC and MSVC-compatible environments are treated as environments where `__int128` is not used as a regular xer feature.

`XER_HAS_INT128` was added so that 128-bit integer paths are enabled only when `__int128` can be used safely. On MSVC and clang-cl environments, relevant code falls back to 64-bit integer based implementations where appropriate.

---

## MSVC-Compatible Identifier Changes

MSVC cl.exe does not accept some symbolic identifiers that contain superscript digits.

ASCII aliases were added for affected unit-related identifiers, including:

```cpp
m2
m3
sec2
sec3
```

These aliases are used on MSVC. Toolchains that accept the symbolic identifiers can continue to use them where available.

---

## Test Diagnostics

Test diagnostics were improved for abnormal process termination.

When a test process terminates because of an uncaught exception, the reason is written to `run.log`. This makes failures easier to diagnose, especially on MSVC, where assertion failures may otherwise appear as runtime abnormal termination without useful output.

---

## Packaging Cleanup

The `vcpkg_installed/` directory generated by vcpkg manifest mode is excluded from Git and release ZIP packages.

Generated build outputs such as `php/build/`, object files, logs, executables, DLLs, import libraries, and temporary files are also kept out of release ZIP packages.

## Compatibility Notes

v1.1.0 mainly improves supported environments and the build/test infrastructure.

No major breaking API changes are intended. The main source-level note is that MSVC support uses ASCII aliases for some unit-related identifiers that contain superscript digits on other toolchains.
