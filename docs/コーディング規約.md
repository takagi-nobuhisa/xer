# コーディング規約

## 目的

本規約は、XER のソースコードにおける可読性、一貫性、保守性を確保することを目的とする。

本規約では、箸の上げ下げのような細かな書式規則を過度に文章化しない。  
書式のうち自動整形で機械的に統一できる事項は、可能な限り clang-format および `.editorconfig` に委ねる。

## 書式に関する基本方針

- 書式の基本スタイルは LLVM スタイルとする。
- 字下げにはタブではなく空白を用いる。
- ソースファイルは BOM 付き UTF-8 とする。
- 改行コード、文字コード、末尾改行、行末空白などは `.editorconfig` で統一する。
- 波括弧の位置、空白の挿入位置、1 行の長さなどは clang-format によって統一する。
- 自動整形で統一可能な事項については、原則として個別の文章規定を設けない。

## 識別子の命名規則

- 識別子の基本は snake case とする。
- 本規約で snake case と記載した場合は、lower snake case を意味する。
- ここまでに個別規則を定めていない識別子も、すべて snake case とする。
- C 言語または PHP の既存関数を移植する場合は、原則としてオリジナルの名前を踏襲する。
- 日本語識別子は採用しない。

## 型名に関する規則

- 型名およびクラス名も snake case とする。
- スカラ型の型名には `_t` をサフィックスとして付ける。

## マクロに関する規則

- マクロは原則として `XER_` をプレフィックスとする大文字表記とする。
- ただし、`xer_assert` のように関数と同様の構文で用いるマクロは小文字表記を用いる。

## 内部実装に関する規則

- 内部実装専用であることを表すために、識別子へ `_impl` などの接尾辞は付けない。
- 内部実装用の識別子は `xer::detail` 名前空間に配置する。

## コメントに関する規則

- コメントは英語で記述する。
- API の説明コメントは Doxygen 形式で記述する。
- Doxygen コメントの記法は JavaDoc 形式を用いる。
- 単なるコードの言い換えではなく、意図、前提、制約、注意点が必要な箇所にコメントを書く。
- 自明な処理に対して過剰なコメントは付けない。

## 運用方針

- 書式上の揺れはレビューで吸収するのではなく、できる限り自動整形で吸収する。
- 規約本文では、XER の設計方針や実装上の注意に関わる事項を優先して定める。
- 規約の追加・修正を行う場合も、まず自動化で解決できるかどうかを検討する。
