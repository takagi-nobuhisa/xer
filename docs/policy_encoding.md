# Character Encoding Conversion Policy

## Supported Character Encodings

The character encodings handled by XER are limited to the following four:

- CP932
- UTF-8
- UTF-16
- UTF-32

---

## Basic Policy for Conversion Methods

Low-level character encoding conversion is implemented according to the following policy:

- **CP932 ⇔ UTF-16** is converted by **table lookup** from arrays
- **UTF-8 ⇔ UTF-32** is converted by **calculation**
- **UTF-16 ⇔ UTF-32** is converted by **calculation**

The conversion tables for CP932 and UTF-16 are **generated by PHP**.

---

## Packed Representations

Low-level processing uses internally packed integer values.  
These are strictly **internal representations** and are not normally intended to be handled directly by users.

### CP932

CP932 is represented by `std::uint16_t`.

- The first byte is placed in the lower 8 bits
- The second byte is placed in the upper 8 bits

For a single-byte character, only the lower 8 bits are used.

### UTF-8

UTF-8 is represented by `std::uint32_t`.

- Bytes are placed from the first byte onward, starting from the lower side

This is done to simplify the logic when expanding into arrays.

### UTF-16

UTF-16 is represented by `std::uint32_t`.

- When a surrogate pair is used:
  - the high surrogate is placed in the lower 16 bits
  - the low surrogate is placed in the upper 16 bits
- When a surrogate pair is not used:
  - only the lower 16 bits are used

---

## Basic Policy for Low-Level Functions

Low-level functions are implemented using the packed representations and conversion rules defined above.  
High-level functions that handle whole strings are built on top of them.

### Nature of Low-Level Functions

- Low-level functions operate on a **single Unicode character**
- Character-level input and output are unified to **`char32_t`**
- Low-level functions that handle only packed integer values are **`constexpr`**

### Return Value on Failure

In low-level functions, failure is represented by:

- a value obtained by **casting `-1` to the result type**

Examples:

- if the function returns `char32_t`, use `static_cast<char32_t>(-1)`
- if the function returns `std::uint32_t`, use `static_cast<std::uint32_t>(-1)`
- if the function returns `std::int32_t`, use `-1`

### Low-Level Conversion for CP932

Conversion from CP932 to UTF-16 only handles BMP characters, but the return type should still be **`char32_t`**.  
The result of conversion from UTF-16 to CP932 may be returned as **`std::int32_t`** so that the failure value `-1` is easy to handle.

---

## Relationship Between Public API and Internal Implementation

Low-level functions that handle packed integer values may be public by design.  
However, because they can easily lead to misuse if handled casually, they are not intended for ordinary use.

### Basic Policy for User-Facing APIs

- UTF-8 is handled as a **string** of `char8_t`
- CP932 is handled as a **string** of `unsigned char`
- Users are not generally expected to manipulate packed integer values directly

### Position of Low-Level Functions

- They are **made public** for cases where they are truly needed
- However, they are intended mainly for **advanced users**
- Ordinary use should prefer the high-level functions

---

## Namespace

Normal APIs and low-level APIs belong to different categories, so their namespaces are separated.

- normal APIs use `xer`
- low-level APIs use `xer::advanced`

For the namespace of low-level APIs, `advanced` is more appropriate than a negative name.

---

## Naming Policy for Low-Level Functions

Low-level functions should use descriptive names that clearly indicate what they accept and what they return.

Examples:

- `xer::advanced::utf32_to_packed_utf8`
- `xer::advanced::packed_utf8_to_utf32`
- `xer::advanced::utf32_to_packed_utf16`
- `xer::advanced::packed_utf16_to_utf32`
- `xer::advanced::utf32_to_packed_cp932`
- `xer::advanced::packed_cp932_to_utf32`

Including `packed_` in the name makes it explicit that these are low-level functions rather than ordinary string conversion functions.

---

## Grapheme Clusters

Separate from character encoding conversion itself, high-level operations such as “extract one character” need to take **grapheme clusters** into account.

### Separation of Units

- the unit of low-level functions is a **code point**
- the unit of a high-level “character” is a **grapheme cluster**

These two must be clearly separated in the design.

### Basic Policy

- low-level functions in `xer::advanced` operate strictly on a **code point basis**
- operations in normal APIs that mean “extract one character” should consider **grapheme clusters**

### Scope

A complete implementation of grapheme clusters is complex, so the possible use of **ICU** may be considered when needed.  
However, the details should be settled later, at the point where functions that actually use ICU are designed.

At present, the following level of organization is sufficient:

- do not force XER alone to carry complex grapheme cluster handling
- delegate to ICU when necessary
- do not go further into detail in this policy document

---

## Note on Target Platforms

Windows 11 or later is the supported baseline on the Windows side.

---

## Error Policy

When a high-level character encoding conversion function detects failure by means of XER’s own UTF-8 / UTF-16 / UTF-32 / CP932 conversion or validity checking, it may use `error_t::encoding_error`.

On the other hand, if an external implementation actually returned `EILSEQ`, that case may continue to use `error_t::ilseq`.

In other words:

- failures detected by XER’s own encoding logic may use `encoding_error`
- failures that directly reflect the behavior of an external implementation may use `ilseq`

This distinction is based on whether the error is conceptually an XER-side encoding error or a direct reflection of a system-level `EILSEQ`.

---

## Summary

- XER supports only CP932, UTF-8, UTF-16, and UTF-32
- CP932 ⇔ UTF-16 conversion is performed by table lookup
- UTF-8 ⇔ UTF-32 and UTF-16 ⇔ UTF-32 conversion are performed by calculation
- CP932 ⇔ UTF-16 conversion tables are generated by PHP
- low-level processing uses packed integer representations internally
- low-level character conversion is unified around `char32_t`
- low-level packed-value functions may be public, but are primarily for advanced users
- normal APIs use `xer`, while low-level APIs use `xer::advanced`
- low-level function names should explicitly indicate packed representations
- grapheme clusters are a separate concern from encoding conversion itself
- complex grapheme cluster handling may be delegated to ICU when needed
- XER-side encoding failures may use `error_t::encoding_error`
- direct reflections of external `EILSEQ` may use `error_t::ilseq`
