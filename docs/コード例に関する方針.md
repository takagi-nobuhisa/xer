# コード例に関する方針

## 概要

XER では、利用者向けのコード例を `examples/` ディレクトリに配置する。

コード例の目的は、単にコンパイルが通る断片を示すことではなく、**利用者が実際にそのまま試せる最小限かつ自然な使用例**を提示することである。

そのため、コード例は次の性質を満たすものとする。

- 単独でコンパイルできる
- 単独で実行できる
- 想定する挙動が読者に分かる
- XER の公開 API の自然な使い方を示す
- 自動実行により動作確認できる

コード例はテストプログラムとは目的が異なる。
テストプログラムが実装の正しさを検証するためのものであるのに対し、コード例は**利用者に使い方を示すためのものである**。

---

## 配置方針

### 配置場所

コード例のソースファイルは `examples/` ディレクトリに配置する。

`examples/` は、`docs/` や `php/` と同じ階層に置く。

### 対象ファイル

コード例は、原則として `.cpp` ファイルとして配置する。

初期段階では、1 ファイルごとに 1 個の独立したコード例を表す構成を基本とする。

---

## コード例の目的

コード例は、少なくとも次のいずれかを満たすものとする。

- 公開 API の基本的な使い方を示す
- よく使う典型的な呼び出し方を示す
- エラー処理を含む自然な利用例を示す
- リファレンスマニュアルやチュートリアルから参照しやすい最小例を示す

一方で、次のようなものはコード例の主目的としない。

- 実装内部の網羅的検証
- 境界条件の徹底的な確認
- 不正使用を前提とした異常系試験
- `assert` 系マクロを多用したテストコードの提示

これらは、原則として `tests/` 側で扱う。

---

## 単独コンパイル・単独実行

### 基本方針

各コード例のソースファイルは、単独でコンパイルおよび実行できなければならない。

ここでいう単独とは、少なくとも次を意味する。

- 他の example ソースファイルに依存しない
- 1 ファイルだけを指定してコンパイルできる
- 実行に必要な最小限の処理がそのファイル内に含まれている

### `main` 関数

コード例のソースファイルは、原則として `main` 関数を含む。

```cpp
auto main() -> int;
````

返却値の型は、XER 全体の方針に合わせて後置形式で記述する。

### 終了コード

コード例は、成功時に `0` を返し、失敗時に非 0 を返すものとする。

最小限の例では、失敗時に `return 1;` としてよい。

---

## 使用する API の方針

### 公開 API を優先する

コード例では、原則として `xer/` 直下の公開ヘッダにある API のみを用いる。

`xer/bits/` 配下の内部実装用 API は、コード例では原則として直接使用しない。

### 自然な書き方を優先する

コード例は、ライブラリ利用者がそのまま真似しやすい自然な書き方を優先する。

したがって、コード例のためだけに不自然な型明示や回避策を書くことは避ける。

もし自然な書き方ができない場合は、コード例側で無理に回避するのではなく、公開 API 側の設計改善を検討すべきである。

### `xer::result` の扱い

通常の公開 API は、`xer::result` を引数として受け取らないという XER 全体の方針に従う。

したがって、コード例でも `xer::result` を保持している場合は、呼び出し側で成功値を明示的に取り出してから次の関数へ渡す。

コード例は、この明示的なエラー処理の流れを自然に示すものとする。

---

## 入出力の方針

### 基本方針

コード例で文字列や文字を表示する場合は、原則として `std::cout` ではなく `xer/stdio.h` の公開 API を用いる。

これは、XER の文字列処理およびテキスト入出力の方針に合わせるためである。

### 理由

XER では、文字列は UTF-8 の `char8_t` 系を基本とし、文字は必要に応じて `char32_t` を用いる。
これらは `std::cout` と素直に相性が良いとは限らない。

そのため、コード例では XER 自身のテキスト入出力 API を用いる方が自然である。

### 推奨する関数

コード例における標準出力への表示では、少なくとも次のような関数の利用を推奨する。

* `xer::puts`
* `xer::fputs`
* `xer::putchar`
* `xer::fputc`

### 標準入力

入力を伴うコード例を作成してもよいが、初期段階では、できる限り入力不要の例を優先する。

入力を伴う例を導入する場合は、PHP スクリプト側で標準入力を制御できることを前提とする。

---

## コメントの方針

### コメントは英語で書く

コード例中のコメントは、XER 全体のコーディング規約に従い、英語で記述する。

### コメントの目的

コード例のコメントは、単なるコードの言い換えではなく、次の情報を読者へ伝えるために用いる。

* この例が何を示すか
* 入力と出力の関係
* 注意すべき前提
* 期待される結果

### 過剰な説明は避ける

コード例は短く読みやすいことが重要であるため、冗長な説明を埋め込みすぎない。

詳細な解説は、必要に応じてドキュメント本文側で行う。

---

## マーカーに関する方針

### 目的

コード例のソースファイルには、後で機械的に抽出しやすいように、コメントによる明示的なマーカーを埋め込む。

このマーカーは、少なくとも次の用途を想定する。

* ドキュメント生成時の抽出
* コード例単位での識別
* 将来的な自動加工や検証

### 必須マーカー

各コード例ファイルには、少なくとも次の 2 つのマーカーを含める。

* `XER_EXAMPLE_BEGIN`
* `XER_EXAMPLE_END`

書式は次のとおりとする。

```cpp
// XER_EXAMPLE_BEGIN: <example_id>
...
// XER_EXAMPLE_END: <example_id>
```

### `example_id`

`<example_id>` には、そのコード例を識別するための識別子を記述する。

識別子は次の条件を満たすものとする。

* 英小文字、数字、アンダースコアを基本とする
* 空白を含めない
* BEGIN と END で完全一致させる
* 内容を見て意味が分かる名前にする

例:

* `trim_basic`
* `json_decode_simple`
* `path_join_basic`

### 位置

`XER_EXAMPLE_BEGIN` は、原則としてファイル先頭付近のコメントブロックの最初に置く。

`XER_EXAMPLE_END` は、原則としてファイル末尾付近に置く。

### 1 ファイル 1 例

初期段階では、1 ファイルにつき 1 個のコード例とし、1 組の BEGIN/END マーカーだけを置く。

1 ファイル中に複数例を埋め込む方式は、少なくとも初期段階では採用しない。

---

## 期待出力の記述

### 目的

コード例には、利用者が目視で期待動作を理解できるように、期待出力をコメントとして明記する。

これは、将来的な機械検証用メタデータとは別目的である。

### 書式

コード例の冒頭コメントには、必要に応じて `Expected output:` を記述する。

例:

```cpp
// XER_EXAMPLE_BEGIN: trim_basic
//
// This example trims leading and trailing ASCII spaces from a UTF-8 string.
//
// Expected output:
// hello
```

### 方針

* `Expected output:` は人間向け説明として記述する
* 実際に表示される内容が分かるよう、簡潔に書く
* 複数行出力する場合は、その各行をコメントとして並べる
* 出力が環境依存になる例では、無理に固定しない

### 入力がある場合

入力を伴うコード例では、必要に応じて `Input:` もコメントで示してよい。

ただし、初期段階では入力不要のコード例を優先する。

---

## コード例とテストプログラムの違い

### テストプログラムではない

コード例は、原則としてテストプログラムではない。

そのため、`tests/` にあるような詳細な検証や、多数の境界値確認をコード例へ持ち込まない。

### `assert` 系マクロの扱い

コード例では、原則として `xer_assert` 系マクロを使用しない。

理由は次のとおりである。

* 利用者向けの読みやすさを損ねやすい
* テストコードに見えやすい
* 本来の API 利用例としての自然さが薄れる

ただし、ある公開 API の典型的利用法として `xer_assert` 自体を示したい場合に限り、そのための専用コード例を設けてもよい。

---

## エラー処理の書き方

### 基本方針

コード例では、エラーを無視せず、最低限の明示的なエラー処理を書く。

### 推奨する形

最小限のコード例では、次のような簡潔な形でよい。

```cpp
if (!result.has_value()) {
    return 1;
}
```

### 過剰な分岐は避ける

コード例の主目的は利用法の提示であるため、エラーの詳細表示や複雑な復旧処理までは原則として書かない。

必要であれば、別のコード例や本文ドキュメントで説明する。

---

## ファイル名に関する方針

### 基本方針

コード例のファイル名は、内容が分かる lower snake case とする。

### 接頭辞

初期段階では、`example_` を接頭辞として付ける。

例:

* `example_trim.cpp`
* `example_json_decode.cpp`
* `example_path_join.cpp`

### 一意性

ファイル名だけを見ても、おおよその題材が分かるようにする。

---

## 自動実行との関係

### 基本方針

`examples/` に配置されたコード例は、PHP スクリプトにより自動的にコンパイルおよび実行できるものとする。

### 対象

少なくとも初期段階では、`php/run_tests.php` により `examples/` を対象として指定したときに、各コード例がビルドおよび実行できることを確認できるようにする。

### 成功条件

初期段階での成功条件は、少なくとも次のとおりとする。

* コンパイルに成功する
* 実行に成功する
* 異常終了しない

### 将来的拡張

必要に応じて、将来的には以下を導入してよい。

* 標準出力の期待値比較
* 標準入力の自動供給
* マーカーからの自動抽出
* ドキュメントへの自動埋め込み

ただし、初期段階ではそこまでを必須としない。

---

## ドキュメントとの関係

### 基本方針

コード例は、将来的にリファレンスマニュアルやチュートリアルから参照または転載しやすい形で記述する。

### 例専用の細工を避ける

ドキュメント抽出専用の複雑な細工や、可読性を損なう特殊記法は避ける。

必要最小限のマーカーとコメントだけを用い、ソースファイルそのものが読みやすい状態を保つ。

### コード例を正本とする

可能な限り、ドキュメント中へ手で再記述したコード片ではなく、`examples/` に置かれたコード例を正本とすることを目指す。

これにより、実際にコンパイル・実行確認された例をドキュメントへ反映しやすくなる。

---

## 推奨テンプレート

コード例ファイルは、少なくとも次のような構成を推奨する。

```cpp
// XER_EXAMPLE_BEGIN: trim_basic
//
// This example trims leading and trailing ASCII spaces from a UTF-8 string.
//
// Expected output:
// hello

#include <xer/stdio.h>
#include <xer/string.h>

auto main() -> int
{
    const auto trimmed = xer::trim_view(std::u8string_view(u8"  hello  "));
    if (!trimmed.has_value()) {
        return 1;
    }

    if (!xer::puts(*trimmed).has_value()) {
        return 1;
    }

    return 0;
}

// XER_EXAMPLE_END: trim_basic
```

具体的な API 名や入力値は例ごとに変わってよいが、次の点はできる限り維持する。

* BEGIN/END マーカー
* 英語コメント
* `Expected output:`
* 単独で動く `main`
* 最小限の明示的エラー処理

---

## まとめ

* コード例は `examples/` に配置する
* 各ファイルは単独でコンパイルおよび実行できるものとする
* 初期段階では 1 ファイル 1 例を原則とする
* 公開 API の自然な使い方を示すことを優先する
* コメントは英語で書く
* 文字・文字列の表示には原則として `xer/stdio.h` を用いる
* `XER_EXAMPLE_BEGIN` / `XER_EXAMPLE_END` を必須とする
* `Expected output:` を人間向けコメントとして記述する
* `assert` 系マクロは原則として examples では使わない
* `examples/` のコード例は PHP スクリプトで自動コンパイル・実行確認できるようにする
* 将来的には、コード例をドキュメントの正本として扱える形を目指す
