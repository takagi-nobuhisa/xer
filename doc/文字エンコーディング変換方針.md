# 文字エンコーディング変換方針

## 対象とする文字エンコーディング

xer で扱う文字エンコーディングは、以下の 4 種類に限定する。

- CP932
- UTF-8
- UTF-16
- UTF-32

---

## 変換方式の基本方針

文字エンコーディング変換の低水準処理は、以下の方針で実装する。

- **CP932 ⇔ UTF-16** は、配列からの**表引き**で変換する
- **UTF-8 ⇔ UTF-32** は、**計算**で変換する
- **UTF-16 ⇔ UTF-32** は、**計算**で変換する

CP932 と UTF-16 の変換テーブルは、**PHP で生成**する。

---

## packed 表現

低水準処理では、内部で packed された整数値を用いる。  
これはあくまでも**内部表現**であり、ユーザーが通常直接扱うことは想定しない。

### CP932

CP932 は `std::uint16_t` で表現する。

- 第1バイトを下位 8 bit に配置する
- 第2バイトを上位 8 bit に配置する

1バイト文字の場合は、下位 8 bit のみを用いる。

### UTF-8

UTF-8 は `std::uint32_t` で表現する。

- 第1バイトから順に、下位側から配置する

これは、配列に展開するときのロジックを簡単にするためである。

### UTF-16

UTF-16 は `std::uint32_t` で表現する。

- サロゲートペアを使う場合:
  - 上位サロゲートを下位 16 bit に配置する
  - 下位サロゲートを上位 16 bit に配置する
- サロゲートペアを使わない場合:
  - 下位 16 bit のみを用いる

---

## 低水準関数の基本方針

低水準関数は、先に定義した packed 表現や変換処理を利用して実装する。  
その上に、文字列全体を扱う高水準関数を構築する。

### 低水準関数の性格

- 低水準関数は **Unicode 文字単位**で扱う
- 文字単位での入出力は **`char32_t`** に統一する
- packed された整数値だけを扱う低水準関数は **`constexpr`** とする

### 失敗時の返却値

低水準関数では、失敗を

- **`-1` を結果型にキャストした値**

で表す。

例:

- `char32_t` を返す関数なら `static_cast<char32_t>(-1)`
- `std::uint32_t` を返す関数なら `static_cast<std::uint32_t>(-1)`
- `std::int32_t` を返す関数なら `-1`

### CP932 の低水準変換

CP932 から UTF-16 への変換は BMP しか扱わないが、返却値は **`char32_t`** とする。  
UTF-16 から CP932 への変換結果は、失敗値 `-1` を扱いやすくするため、**`std::int32_t`** で返してよい。

---

## 公開 API と内部実装の関係

packed された整数値を扱う低水準関数は、設計としては公開してもよい。  
ただし、これらはユーザーが雑に扱うと事故につながりやすいため、通常利用はあまり推奨しない。

### ユーザー向け API の基本

- UTF-8 は `char8_t` 系の**文字列**として扱う
- CP932 は `unsigned char` 系の**文字列**として扱う
- ユーザーが packed 整数値を直接扱うことは、基本的に想定しない

### 低水準関数の位置づけ

- どうしても必要な場合のために**公開はする**
- ただし基本的には**上級者向け**とする
- 通常利用は高水準関数を推奨する

---

## 名前空間

通常の API と低水準 API ではカテゴリが異なるため、名前空間を分離する。

- 通常 API は `xer`
- 低水準 API は `xer::advanced`

低水準 API 用の名前空間としては、ネガティブな名前よりも `advanced` の方が適切である。

---

## 低水準関数の命名方針

低水準関数は、何を受け取り何を返すかが明確に分かる、説明的な名前にする。

例:

- `xer::advanced::utf32_to_packed_utf8`
- `xer::advanced::packed_utf8_to_utf32`
- `xer::advanced::utf32_to_packed_utf16`
- `xer::advanced::packed_utf16_to_utf32`
- `xer::advanced::utf32_to_packed_cp932`
- `xer::advanced::packed_cp932_to_utf32`

`packed_` を名前に含めることで、通常の文字列変換関数ではなく、低水準関数であることを明示する。

---

## 書記素クラスタ

文字コード変換そのものとは別に、「1文字を取り出す」という高水準処理では **書記素クラスタ** を考慮する必要がある。

### 単位の分離

- 低水準関数の単位は **コードポイント**
- 高水準の「1文字」は **書記素クラスタ**

この 2 つは明確に分けて設計する。

### 基本方針

- `xer::advanced` の低水準関数は、あくまで**コードポイント単位**で処理する
- 通常 API における「1文字を取り出す」処理では、**書記素クラスタ**を意識する

### 対応範囲

書記素クラスタの完全実装は複雑であるため、必要に応じて **ICU** の利用を視野に入れる。  
ただし、ICU を実際に利用する関数を作る段階で、あらためて詳細を詰める。

現時点では、以下のような整理でよい。

- 複雑な書記素クラスタ処理は xer 単体で無理に背負わない
- 必要に応じて ICU に委譲する
- 今回の議論では、それ以上の詳細には立ち入らない

---

## 対象プラットフォームに関する補足

Windows 側は **Windows 11 以降**を対象とする。  
Windows 10 はすでにセキュリティサポートが終了しているため、考慮対象から外してよい。

---


## エラー方針

文字エンコーディング変換の高水準関数が、XER 自前の UTF-8 / UTF-16 / UTF-32 / CP932 変換や妥当性検査により失敗を検出した場合は、`error_t::encoding_error` を用いてよい。
一方、外部の処理系が実際に `EILSEQ` を返したことを表したい場合に限り、`error_t::ilseq` を用いる。
