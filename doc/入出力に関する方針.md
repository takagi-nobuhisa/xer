# 入出力に関する方針

## 基本方針

XER の入出力は、C 標準ライブラリの `FILE` 系関数を土台とした設計とする。  
ただし、公開 API は C++ ライブラリとして再設計し、C と完全互換の形は目指さない。

- 低水準基盤は `FILE` 型を用いた入出力を利用する
- `iostream` は使用しない
- 高水準な入出力関数は、この低水準基盤の上に構築する

使用する主な低水準関数は以下とする。

- `fopen`
- `fclose`
- `fseek`
- `ftell`
- `setvbuf`
- `feof`
- `ferror`
- `fread`
- `fwrite`

## 公開型

公開 API では `FILE` をそのまま公開せず、用途に応じて以下の型を用いる。

- `binary_stream`
- `text_stream`

両者は明確に分離し、同一型の中でモード切り替えを行う設計にはしない。

これらの型により、以下のような対象を統一的に扱う。

- ファイル
- メモリ上のデータ
- 文字列
- 標準入力
- 標準出力
- 標準エラー出力

## `binary_stream` / `text_stream` の設計方針

### move-only

`binary_stream` および `text_stream` はコピー禁止、ムーブのみ可能とする。

### RAII

これらのストリーム型は RAII により open / close を管理する。

ここでいう RAII は、ストリームが保持する内部状態の寿命管理を含む。  
したがって、open 系関数で確保した内部状態は、`fclose` またはデストラクタにより解放される。

### デストラクタ

デストラクタでは自動クローズを行う。

- open 系関数で作成された内部状態は、デストラクタで `close` 相当の処理を行ったうえで破棄する
- close が失敗しても、その失敗はデストラクタからは通知しない
- デストラクタ後に同じストリームオブジェクトを再利用することは考えない

### 明示クローズ

`fclose` は明示操作として提供する。

- `fclose` 後、そのストリームは成功失敗にかかわらずクローズ済みとする
- 二重クローズは no-op とする
- 同じストリームの再オープンは禁止する
- 再利用したい場合は、新たに open 系関数で開いたストリームをムーブする

### クローズ済み状態

クローズ済みのストリームに対する操作は失敗とする。  
現時点ではその失敗は `error_t::runtime_error` に正規化してよい。

## 内部実装方針

各ストリーム型は、内部で以下を保持する軽量なハンドル型とする。

- 不透明ハンドル値
- close / read / write / seek / tell などの関数ポインタ群

不透明ハンドル値には `std::uintptr_t` を用いる。  
このハンドルは、内部状態オブジェクトまたは `FILE*` などを `reinterpret_cast` で格納するためのものとする。

### `binary_stream`

`binary_stream` は少なくとも以下の内部関数ポインタを保持する。

- `close`
- `read`
- `write`
- `seek`
- `tell`

### `text_stream`

`text_stream` は少なくとも以下の内部関数ポインタを保持する。

- `close`
- `read`
- `write`
- `getpos`
- `setpos`
- `seek_end`

ここでいう `read` / `write` はテキストの文字入出力処理用の入口であり、  
現時点では `fopen` 自体はそれらを完成させる責務を持たず、`fgetc` / `fputc` 相当などの上位関数で実処理する。

### 既定のエラー関数

空ストリームや未対応操作に対しては、`nullptr` を用いず、各シグニチャごとに用意したエラー関数を設定する。  
これにより、呼び出し側で `nullptr` 判定を行わずに済む。

## オープン関数

### ファイル

ファイルを開く関数は、バイナリ用とテキスト用を分ける。

- バイナリ用 open 関数は `binary_stream` を返す
- テキスト用 open 関数は `text_stream` を返す

シグニチャは次のようにする。

- `std::expected<binary_stream, error<void>> fopen(const path& filename, const char* mode) noexcept;`
- `std::expected<text_stream, error<void>> fopen(const path& filename, const char* mode, encoding_t encoding) noexcept;`

### メモリ

メモリを開く関数は用途ごとに分離する。

- `memopen` はバイナリ用とし、`binary_stream` を返す
- `stropen` はテキスト用とし、`text_stream` を返す

現時点では、これらは外部実体を借用する形で扱う。

- `memopen(std::span<std::byte>, const char* mode)`
- `stropen(std::u8string_view, const char* mode)`
- `stropen(std::u8string&, const char* mode)`

ファイルとの対称性を重視し、open 系関数がコンテナ所有権を引き受ける形は採らない。

## モード指定

モード指定は C と同様、`char` 文字列で行う。

例:

- `"r"`
- `"w"`
- `"a"`
- `"rb"`
- `"wb"`
- `"r+"`

モード文字列は制御用トークンであり、テキストデータそのものではないため、`char8_t` 系は用いない。

排他モードである `"x"` 系は現時点では非対応とする。

### `b` / `t` の扱い

- バイナリ open では `b` は省略可、`t` は不正とする
- テキスト open では `t` は省略可、`b` は不正とする
- `b` / `t` の位置の揺れは許容する
- open 種別そのものはオーバーロードで決まり、`b` / `t` は補助的な検証用トークンとして扱う

### 内部での正規化

モード文字列は内部で次の列挙値に正規化する。

- `r`
- `rp`
- `w`
- `wp`
- `a`
- `ap`
- `error`

ここで `p` は `+` を表す内部表現である。

この正規化結果は open 時の一時情報であり、ストリーム内部には保持しない。  
open 後の振る舞いは、関数ポインタと内部状態の組み合わせで決定する。

## バイナリストリーム方針

### 基本方針

バイナリデータは `std::byte` で扱う。

### 関数

- `fread`
- `fwrite`

は同名で再実装する。

1 バイト単位の入出力には以下を用いる。

- `fgetb`
- `fputb`

`fgetc` / `fputc` は文字向けの名称であるため、バイナリ 1 バイト入出力には用いない。

### EOF の扱い

`fgetb` において EOF に到達した場合、その 1 バイト入力操作は失敗であるとみなし、エラー扱いとする。

### 位置操作

`binary_stream` では通常の byte 単位の seek / tell を提供する。

### 内部状態

`binary_stream` の内部状態は、少なくとも以下のいずれかを保持できるものとする。

- `FILE*` を持つファイルソース
- バイト列領域と現在位置を持つメモリソース

書き込み可否、読み込み可否、append 動作は内部状態に保持してよい。

## テキストストリーム方針

### 基本方針

テキストモードでは、UTF-8 と CP932 を扱う。  
ロケールには依存しない。

テキスト入出力の外部エンコーディングは以下に限定する。

- UTF-8
- CP932

内部で扱う値は以下のように正規化する。

- 1 文字入力は `char32_t`
- 文字列入力は UTF-8 の `char8_t` 文字列

入力元の実エンコーディングが UTF-8 であっても CP932 であっても、上位 API から見た結果表現は共通とする。

### テキスト入力時のエンコーディング指定

テキスト入力時のエンコーディング指定は以下とする。

- `utf8`
- `cp932`
- `auto_detect`

`auto_detect` 指定時のみ自動判別を行う。

### テキスト出力時のエンコーディング指定

現時点のテキスト open 関数で受けるエンコーディング指定も以下の 3 種類とする。

- `utf8`
- `cp932`
- `auto_detect`

ただし、`auto_detect` は入力時専用とし、書き込み系モードでは不正とする。

BOM の有無や UTF-8N 相当の区別は、現時点ではまだ公開仕様として確定していない。  
したがって、現段階の `encoding_t` には含めない。

### `text_stream_state`

`text_stream` のハンドル先には `text_stream_state` を持たせる。  
ここには少なくとも以下を保持する。

- 入出力対象
  - `FILE*`
  - 読み取り専用文字列ビュー
  - 書き込み可能文字列
- 先読みバッファ
  - 1024 バイト
- 確定済みまたは未確定のエンコーディング状態
- `mbstate_t` 相当の途中状態

### 先読みバッファ

自動判別およびマルチバイト文字処理のため、`text_stream_state` は 1024 バイトの先読みバッファを持つ。

### `mbstate_t` 相当の状態

UTF-8 や CP932 の途中状態を扱うため、`text_stream_state` には `mbstate_t` 相当の内部状態を持たせる。  
現時点では、UTF-8 の未完成部分や CP932 の途中状態を保持できればよい。

## テキスト入力の自動判別方針

### 基本方針

`auto_detect` 指定時は、UTF-8 と CP932 の自動判別を行う。  
誤認識の可能性はゼロにはできないため、一定確率で誤認識することは許容する。

### 判定規則

判定は次の優先順位で行う。

1. UTF-8 BOM があれば UTF-8
2. BOM がなく、CP932 の妥当な多バイト文字列が見つかれば CP932
3. それ以外は UTF-8

ただし、ASCII しか観測していない段階では UTF-8 に確定したものとはみなさない。

### 判定の進め方

自動判別は先読み専用の一括処理ではなく、入力の進行に応じて行う。  
ただし、内部では 1 KiB 単位程度のバッファリングを行ってよい。

- 1 KiB 読み込んで判定材料があれば確定する
- ASCII しかなければ未確定のまま次の入力へ進む
- 次の 1 KiB でも判定を継続してよい
- EOF まで確定しなければ UTF-8 とみなす

### 未確定状態の扱い

未確定状態では ASCII は問題なく入力できるものとする。  
ASCII は UTF-8 と CP932 の共通部分であるためである。

非 ASCII バイトが現れた場合にのみ、UTF-8 か CP932 かの判別を進める。

この判別のために未確定状態で保留するバイト数は、高々 1 バイトとする。  
ここでいう保留は UTF-8 文字としての完全性検査ではなく、**UTF-8 と CP932 のどちらとして解釈すべきかの判別**のための保留である。

UTF-8 の 3 バイト文字・4 バイト文字として正しいかどうかは、UTF-8 と確定した後の通常デコード処理で扱う。

### 不正入力の扱い

バッファリングの都合ではなく、本当に多バイト文字の片割れだけが入力された場合はエラーとする。  
UTF-8 と確定した後に UTF-8 として不正な並びが現れた場合もエラーとする。  
CP932 と確定した後に CP932 として不正な並びが現れた場合も同様とする。
